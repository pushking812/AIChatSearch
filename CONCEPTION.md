Краткая концепция приложения для поиска по архиву чатов DeepSeek

1. Цель
Разработать десктопное приложение на Python (3.11+) с графическим интерфейсом на Tkinter для загрузки ZIP-архива с файлом conversation.json, быстрого поиска сообщений по различным критериям и удобного просмотра/редактирования найденных пар «запрос–ответ». Ожидаемый объём данных: 50–500 чатов, 10–100 пар сообщений в каждом.

2. Структура данных
conversation.json содержит массив объектов чатов.
Каждый чат имеет поля: id, title, inserted_at, updated_at, mapping (объект с узлами).
Узлы в mapping:
Ключ — строковый идентификатор (например, "1", "2").
Узел содержит id, parent, children, message.
message присутствует только в узлах, соответствующих сообщениям, и содержит:
inserted_at — дата/время сообщения.
fragments — массив фрагментов. Упрощённо, первый фрагмент с type: "REQUEST" или "RESPONSE" содержит основной текст в поле content.
Сообщения идут строго парами «запрос → ответ» в порядке возрастания inserted_at. Для получения линейной последовательности достаточно собрать все узлы с message (они уже упорядочены по времени). Древовидная структура не создаёт ветвлений и на навигацию не влияет.

3. Основные компоненты GUI
Окно разделено на две основные панели: левая – навигация по чатам и настройки поиска, правая – результаты и просмотр.

3.1 Левая панель
3.1.1 Верхняя часть (поиск)
Выпадающий список (Combobox) для выбора области поиска:
Название чата
Запрос пользователя (фрагменты с type: "REQUEST")
Ответ чат-бота (фрагменты с type: "RESPONSE")
Поле ввода текста (Entry) для поискового запроса.
Флажок «Искать по мере ввода» (опция автоматического поиска при изменении текста).
Кнопка «Найти» для запуска поиска.

3.1.2 Нижняя часть (список чатов)
Поле фильтрации чатов по названию (Entry) – при вводе текст динамически фильтрует список ниже.
Список всех загруженных чатов (Listbox с поддержкой множественного выбора с Ctrl/Shift).
Кнопки «Выбрать все» и «Снять выделение» для управления выделением.
Поиск осуществляется по чатам, прошедшим фильтр и выделенным в списке. Если выделение пусто, поиск идёт по всем отфильтрованным чатам.

3.2 Правая панель
3.2.1 Верхняя часть (результаты поиска)
Используется Treeview для группировки найденных пар по чатам.
Корневые узлы — названия чатов с датой создания и обновления (например, «Чат: Название (2025-11-14 03:50)»).
Дочерние узлы — каждая найденная пара сообщений. Отображают:
Индекс пары внутри чата (номер из JSON, например «1»).
Дату/время создания ответа (или запроса, если ответ ещё не получен).
Превью запроса (50–100 символов, если поиск по названию или по ответу — первые символы; если поиск по запросу — с выделением контекста вокруг искомого).
Превью ответа (аналогично).
При клике на узел пары в нижней панели отображается полный контекст.

3.2.2 Нижняя часть (просмотр/редактирование)
Два текстовых поля (Text) с прокруткой:
Верхнее поле — полный текст запроса.
Нижнее поле — полный текст ответа.
В обоих полях искомый текст (если поиск вёлся по соответствующему полю) подсвечивается (например, жёлтым фоном).
Поля доступны для редактирования.
Кнопки переключения:
«Предыдущая пара» / «Следующая пара» — перемещение по парам внутри текущего чата, начиная с выбранной пары.
Кнопка «Сохранить изменения» (базовая функциональность) — сохраняет отредактированные тексты в текущую сессию (в память).
Расширенная функциональность (будет реализована позже):
«Сохранить в архив» — перезапись исходного conversation.json в том же ZIP-архиве.
«Сохранить как…» — создание новой ZIP-копии с изменённым JSON.

4. Логика работы
Загрузка архива: меню «Файл» → «Открыть архив» (или кнопка). Выбирается ZIP-файл, извлекается conversation.json, парсится, данные сохраняются в памяти. Строится удобная структура: список чатов, каждый содержит упорядоченный список пар сообщений (номер, запрос, ответ, даты).
Построение списка чатов: левый нижний список заполняется названиями. При вводе текста в поле фильтра список динамически обновляется.
Поиск: запускается по кнопке «Найти» или автоматически при вводе (если включена опция). Берётся текст запроса, область поиска, учитываются отфильтрованные и выделенные чаты. Для каждой пары сообщений проверяется:
При поиске по названию: вхождение в title чата.
При поиске по запросу: вхождение в текст запроса.
При поиске по ответу: вхождение в текст ответа.
Результаты группируются по чатам и отображаются в Treeview.
Просмотр пары: при выборе узла в Treeview заполняются текстовые поля: запрос и ответ. Искомый текст подсвечивается.
Навигация по парам: кнопки «Предыдущая/Следующая пара» переключают между парами текущего чата, обновляя поля.
Редактирование: пользователь может изменять текст в полях. Кнопка «Сохранить изменения» обновляет данные в памяти. В будущем будет реализовано сохранение в архив.

5. Технические аспекты
Язык: Python >= 3.11.
Библиотеки: только стандартные: tkinter, zipfile, json, datetime (для парсинга дат).

Структура кода: модули:
model.py — классы для загрузки, хранения и поиска по данным.
gui.py — основное окно и компоненты интерфейса.
utils.py — вспомогательные функции (форматирование дат, подсветка и т.д.).

Оптимизация: при загрузке строятся индексы для быстрого поиска (например, для каждого чата — множество слов или просто in-проверка). Визуальные компоненты используют стандартные списки/деревья без виртуализации (объём данных позволяет).

6. Дополнительные соображения
Подсветка в текстовых полях будет реализована через теги Tkinter.
Сохранение в архив: будет использоваться временная директория для извлечения, модификации JSON и упаковки обратно.
Интернационализация не требуется.

8. Обработка edge-кейсов

В текущей реализации предусмотрена защита от некорректных данных и действий пользователя:

• В model.load_from_zip:
  – пропуск неполных пар (REQUEST без RESPONSE);
  – пропуск узлов без message;
  – проверка соответствия типов REQUEST → RESPONSE;
  – защита от отсутствующего inserted_at;
  – игнорирование некорректных fragments.

• В GUI:
  – защита от пустого выбора в Listbox;
  – защита от обращения к несуществующей паре;
  – очистка текстовых полей перед обновлением;
  – безопасная обработка отмены выбора файла.

Актуализация реализации поиска и подсветки (версия 454+):

• Подсветка искомого текста реализована через кастомный тег Tkinter "search_match".
• Подсветка не зависит от наличия фокуса в текстовом поле.
• При Live-search фокус остаётся в поле ввода поиска.
• При навигации по кнопкам (< >) фокус переводится в соответствующее текстовое поле.
• Выделение применяется асинхронно через after_idle для корректной работы с Treeview.

## Навигация при активном поиске

1. Поиск влияет только на отображаемый список сообщений в TreeView.
2. Кнопки Next / Prev всегда работают в пределах текущего чата.
3. Текущий чат определяется как чат сообщения, выделенного в TreeView.
4. Поиск не изменяет источник данных для навигации.
5. Данное поведение является проектным решением (feature) и не считается дефектом.


### Обработка отсутствия результатов поиска

Если поиск не возвращает совпадений:
- TreeView отображает пустой список.
- Кнопки навигации Next / Prev должны быть заблокированы на уровне GUI.
- Контроллер допускает состояние current_pair_index = None.
- Дополнительная обработка данного edge-case на уровне контроллера не требуется.


### Смена чата при активном поиске

При выборе нового чата:
- search_active сбрасывается в False.
- visible_pairs очищается.
- current_chat обновляется.
- current_pair_index устанавливается в 0, если в чате есть пары, иначе None.

Таким образом, состояние поиска полностью сбрасывается при смене чата.


### Обработка одного результата поиска

Если поиск возвращает ровно одно сообщение:
- TreeView содержит один элемент.
- current_pair_index устанавливается в 0.
- Навигация Next / Prev продолжает работать в пределах current_chat.pairs.
- Такое поведение является feature и не требует синхронизации индекса с позицией найденной пары.
